# Static API Implementation for the Quartz Website

## 1. Purpose

Implement a static, read-only API as part of the Quartz build.

The API will expose the same canonical ICH E6(R3) data used to generate the human-readable website. It must be usable by external AI agents and other software without requiring them to parse rendered HTML.

The API will be hosted through GitHub Pages together with the Quartz website. Because GitHub Pages only serves static files, all API responses must be generated during the build process.

## 2. Core approach

Start with a static API generated by Quartz.

The implementation must provide:

- one JSON resource per guideline section;
- one JSON resource per numbered clause;
- glossary JSON files;
- obligation-register JSON files;
- role-specific collections;
- bulk JSONL downloads;
- stable canonical identifiers;
- content hashes and version metadata;
- an OpenAPI document describing the static endpoints.

All API resources must be generated from the same canonical structured data used by the Markdown and HTML pages.

Do not create a separate manually maintained API dataset.

## 3. Constraints

The API is static and read-only.

It may support HTTP `GET` requests for generated files, but it cannot provide runtime functionality such as:

- dynamic server-side search;
- `POST` requests;
- AI-generated responses;
- claim verification at request time;
- authentication;
- server-side sessions;
- database queries;
- request-specific filtering;
- a continuously running MCP server.

Any feature requiring runtime computation must be implemented later as a separate backend service.

## 4. Canonical source data

Generate the API from canonical project data files such as:

```text
data/
├── documents.json
├── sections.json
├── clauses.json
├── alignments.json
├── glossary.json
├── terminology.json
├── roles.json
├── obligations.json
├── essential-records.json
└── versions.json
```

If `clauses.json` does not yet exist, generate it from the parsed section hierarchy before producing the API files.

The build must fail if required canonical data is missing or invalid.

## 6. URL design

Use stable, predictable URLs.

```text
/api/v1/sections/ich-e6-r3-a1-2.8.json
/api/v1/clauses/ich-e6-r3-a1-2.8.6.json
/api/v1/concepts/tietoon-perustuva-suostumus.json
/api/v1/obligations/INV-IC-001.json
/api/v1/roles/tutkija.json
/api/v1/downloads/sections.jsonl
/openapi.json
```

Do not include source filenames, local file paths, build timestamps, or temporary identifiers in public URLs.

Stable identifiers must remain unchanged when:

- page titles change;
- Markdown files move;
- navigation changes;
- Quartz configuration changes;
- the website receives a new visual design.

```json
  "title_en": "Informed Consent of Trial Participants",
  "text_fi": "...",

  "document_id_fi": "ich-e6-r3-fi-v1",
  "document_id_en": "ich-e6-r3-en-step5",
  "translation_status": "unofficial",
  "authoritative_language": "en",

  "parent_id": "ich-e6-r3-a1-2",
```

- one resource per terminology entry where useful.

Glossary concepts must distinguish between:

- formal ICH glossary entries;
- Finnish translation mappings;
- synonyms;
- abbreviations;

```json
  "definition_en": "...",
  "alternative_labels_fi": [],

  "search_variants_fi": [
    "tietoon perustuvan suostumuksen"
```

## 12. Bulk JSONL downloads

Generate newline-delimited JSON files for efficient agent ingestion.

Required files:

```text
/api/v1/downloads/sections.jsonl
/api/v1/downloads/clauses.jsonl
/api/v1/downloads/glossary.jsonl
/api/v1/downloads/obligations.jsonl
/api/v1/downloads/roles.jsonl
```

Requirements:

- one complete JSON object per line;
- UTF-8 encoding;
- Unicode NFC normalization;
- no surrounding JSON array;
- deterministic record ordering;
- final newline at end of file;
- schema version included in every record or in an accompanying manifest;
- matching SHA-256 checksums.

## 14. Content hashes

Every individual API resource must include a content hash.

Use SHA-256.

The hash must be calculated from a deterministic serialization of the substantive resource content.

Do not include volatile fields such as the build timestamp in the hash input.

Recommended process:

1. remove volatile metadata;
2. sort object keys;
3. serialize consistently as UTF-8 JSON;
4. calculate SHA-256;
5. add the resulting hash to the published resource.

Also generate:

```text
/api/v1/downloads/checksums.sha256
```

This file must list checksums for all bulk downloads and major collection files.

## 15. OpenAPI document

Generate or maintain:

```text
/openapi.json
```

The OpenAPI file must document the static resources as read-only `GET` endpoints.

Minimum documented paths:

```text
/api/v1/manifest.json
/api/v1/documents.json
/api/v1/sections.json
/api/v1/sections/{section_id}.json
/api/v1/clauses/{clause_id}.json
/api/v1/concepts/{concept_id}.json
/api/v1/obligations/{obligation_id}.json
/api/v1/roles/{role_id}.json
/api/v1/downloads/sections.jsonl
/api/v1/downloads/clauses.jsonl
/api/v1/downloads/obligations.jsonl
```

The OpenAPI document must explain that:

- the API is static;
- only `GET` is supported;
- missing resources return the GitHub Pages static-hosting response;
- query parameters are not supported unless implemented through prebuilt files;
- filtering is performed through precomputed collections;
- runtime search is not included.

Define reusable schemas for:

- document;
- section;
- clause;
- glossary concept;
- terminology entry;
- obligation;
- role collection;
- manifest.

## 16. Quartz integration

Implement API generation as one of:

1. a Quartz emitter plugin;
2. a pre-build script executed before the Quartz build;
3. a post-build script that writes additional files into the final output directory.

Preferred approach:

- use a dedicated build script for canonical JSON generation;
- use a Quartz emitter for files that depend on Quartz page metadata;
- ensure one command builds both the website and the API.

Example:

```bash
npm run build
```

The build must:

1. validate canonical data;
2. generate API resources;
3. generate Markdown or validate existing Markdown;
4. build Quartz;
5. copy or emit API files into the final public directory;
6. validate generated URLs and JSON;
7. generate the build report.

## 17. GitHub Actions deployment

The GitHub Actions workflow must deploy both:

- the Quartz website;
- the generated static API files.

The deployed artifact must contain:

```text
index.html
openapi.json
api/v1/...
```

The workflow must fail if:

- the API build fails;
- JSON validation fails;
- required files are missing;
- checksums cannot be generated;
- the Quartz build fails;
- internal API references are broken.

## 18. Validation requirements

Validate every generated JSON file against a JSON Schema.

```text
schemas/
├── document.schema.json
├── section.schema.json
├── clause.schema.json
├── concept.schema.json
├── obligation.schema.json
├── role.schema.json
└── manifest.schema.json
```

Required tests:

- every resource validates against its schema;
- every identifier is unique;
- every referenced identifier exists;
- every canonical URL is valid;
- every API URL is valid;
- every content hash matches the serialized content;
- all JSON files are valid UTF-8;
- all text is Unicode NFC;
- no local `file:///` paths appear;
- no Windows paths appear;
- no unpublished source paths appear;
- section and clause counts match the canonical data;
- JSONL line counts match the manifest;
- OpenAPI validates against the selected OpenAPI version;
- the production Quartz build includes all required API files.

## 19. Deterministic output

The same canonical input must produce the same substantive API output.

Requirements:

- sort collection records by stable identifier;
- sort object keys where deterministic serialization is required;
- avoid random identifiers;
- do not include machine-specific paths;
- keep volatile timestamps outside content-hash calculations;
- use fixed schema and API versions;
- normalize line endings;
- normalize Unicode to NFC.

```text
content/kehittajille-ja-agenteille/index.md
```

It should explain:

- what the static API contains;

- identifier format;
- versioning policy;
- translation status;
- source authority;
- example `GET` requests;
- JSONL download usage;
- checksum verification;

Example:

```bash
curl https://example.org/api/v1/sections/ich-e6-r3-a1-2.8.json
```

## 21. Out of scope for the first version

Do not implement:

- dynamic search endpoints;
- semantic search;
- vector databases;
- `POST /verify-claim`;
- AI question answering;
- MCP server;
- user authentication;
- runtime filtering;
- write operations;
- personalized responses.

These may be added later through a separate service that consumes the same canonical datasets.

## 22. Acceptance criteria

The implementation is complete when:

- one JSON file exists for every guideline section;
- one JSON file exists for every numbered clause;
- glossary and terminology collections are published;
- one JSON file exists for every obligation;
- one JSON collection exists for every controlled role;
- all required JSONL downloads are published;
- every resource has a stable identifier;
- every resource has version metadata;
- every resource has a valid content hash;
- a manifest describes the dataset and record counts;
- checksums are published;
- `openapi.json` documents all static endpoints;
- all JSON files pass schema validation;
- all cross-references resolve;
- the local Quartz build succeeds;
- the GitHub Pages deployment includes the API;
- external clients can retrieve the published JSON files using ordinary HTTP `GET` requests;
- no runtime backend is required.

## 23. Architectural principle

```text
Canonical structured data
        ↓
Static API generator
        ├── Individual JSON resources
        ├── Collection JSON files
        ├── JSONL bulk downloads
        ├── checksums
        └── OpenAPI document
        ↓
Quartz and GitHub Pages deployment
        ├── Human-readable website
        └── Static read-only API
```

The website and API must always be generated from the same canonical data.

Do not allow the human-readable pages and machine-readable resources to become independent or inconsistent publications.
